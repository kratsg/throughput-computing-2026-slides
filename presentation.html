The whole talk in one example

“Where do my TopCPToolkit outputs go?”

Your jobs emit big ntuples and small histograms. Where should each land on the AF?

A generic LLM

“Write the outputs next to your job” — i.e. $HOME. The big ntuples blow the 100 GB /home quota almost immediately.

An LLM that knows the AF

big ntuples → /data (5 TB); small histograms → /home (100 GB). It knows the quotas and layout.

The thesis

The model didn’t get smarter. We gave it facility context. That gap — between a generic assistant and one that knows your storage, scheduler, and data — is the whole game.

This one example is the entire talk, and it's AF-specific on purpose. An analyzer runs TopCPToolkit: large output ntuples, small histograms. Ask a generic model where to write them and it'll say "next to your job" — your home directory — which is only 100 GB. The big ntuples blow that quota almost immediately. An AF-aware agent routes the big ntuples to /data (5 TB) and keeps the small histograms in /home. Same model — it just knows the quotas and layout. Everything else today is variations on "wrap facility knowledge and tools behind the agent."

Grounding — the vocabulary

Four words, one minute

Agent

The decision-maker: a model that picks the next tool to call, reads the result, and repeats. The “brain.”

MCP server

Model Context Protocol: a standard adapter exposing a service (Rucio, AMI, your notebook) as tools and resources any agent can call.

Skill

A reusable bundle of instructions + know-how the agent loads on demand (“how we do a pyhf fit here”).

Agent harness

The app that gives the agent hands: runs its commands, reads/writes files, wires MCPs, enforces permissions. e.g. Claude Code, Codex.

Agent

MCP server

Rucio / AMI / Jupyter / …

Fast, because this room is mixed. Agent = the brain: the model deciding which tool to call next. Harness = the hands: the app that actually runs those commands, touches files, wires up MCPs, enforces permissions — Claude Code, Codex. MCP = the USB-C port exposing a service as tools and resources. Skill = instructions on demand. Don't dwell — these are just the nouns for the rest of the talk.

Write the facility down once, every analyzer's agent inherits it. That's the cheapest, highest-leverage thing we did. Everything else is plumbing to deliver context like this.

What you can use today · Jupyter MCP

Drive your AF notebook from anywhere

An MCP server runs inside your JupyterLab pod. Any agent that reaches it — Claude Code, Claude Desktop, even your phone — can insert_execute_code_cell, read_cell (incl. plots), execute_code in the live kernel, use_notebook.

≠ Jupyter AI (agent inside the notebook server) — which may lose out to the VS Code workflow many users prefer.

claude mcp add jupyter --transport http \
  .../user/<you>/mcp --header "Authorization: Bearer <tok>"

  The convenience/security tradeoff

Servers have finite lifetimes (equitable sharing) → you re-mint URL + token each respawn. Idea: a Keycloak-OIDC reverse-proxy → one token, pick your server in-browser (but one at a time).

My favorite demo. The Jupyter MCP runs inside your JupyterLab pod, so the chat happens in ANY agent that reaches the endpoint — I've driven a live AF notebook from my phone. That's a different bet, and honestly it may lose out — a lot of our users prefer the VS Code workflow over JupyterLab. One idea I'm weighing: a Keycloak-OIDC reverse-proxy so you keep a single token and just pick which server it routes to in the browser — the cost being you can only drive one server at a time.

Why it works

We've seen the what. Now the why — the three architectural ideas that make all of that hang together and, critically, make it portable.

The core insight

The first two blocks are interchangeable — pick any model (or self-host to cut cost), any harness. Every win today came from the third: the managed CLAUDE.md, the marketplace plugins, the Rucio/AMI MCPs, the condor bot’s HTCondor.md.

Self-hosting option for open-weight models: NRP-hosted LLMs (nrp.ai/llms).
If you take one slide home, this one — and I'm dropping the buzzwords to be concrete. Three ingredients. The model — Opus, GPT-5, or a locally-hosted open-weight model, which matters for cost at HL-LHC scale. The harness — Claude Code, Codex, Copilot. Both of those you can swap freely; they're not where our work is. The third block is the facility context and ATLAS tools, and THAT is where every win in section one came from: the CLAUDE.md, the marketplace, the MCPs, the agent's memory file. So: models keep improving on their own, harnesses are a preference, and you can self-host to control cost. Our job — the facility's job — is the context. That's why this is a facility conversation, not just a vendor one.

The core insight · the flip side

No context? It should stay quiet.

  What actually happened

Our cluster agents confidently recommended Lustre/MDS tuning for a filesystem that is actually Ceph — they had zero Ceph context. Plausible, fluent, and wrong.

“If we’re supposed to rely on the agents, they need to be accurate — otherwise they can’t be trusted.” Judith, #analysis-facility
“If the agent has no Ceph context, it shouldn’t make recommendations or ping people — we’d just chase irrelevant suggestions.” Farnaz, same thread

The rule we landed on: if it isn’t grounded in real facility context, it stays silent rather than speculates. Context isn’t just what makes the agent useful — it’s what makes it safe to trust.

The flip side of the thesis, and it's a real scrape. Early on, our cluster-monitoring agents recommended Lustre and MDS tuning for a problem on a filesystem that's actually Ceph. They had no Ceph context, so they pattern-matched to the storage they DID know — confident, fluent, and wrong. The team's reaction matters. Judith: if we're going to rely on these, they have to be accurate or they can't be trusted. Farnaz: if it has no Ceph context, it shouldn't be making recommendations or pinging people — we'll just chase irrelevant noise. Ilija's point: those agents had literally zero Ceph instructions. So the rule we adopted: no grounded context, stay quiet. This is the same thesis from the other side — context isn't only what makes the agent useful, it's what makes it safe to trust. And it's exactly why "where does the knowledge live and who keeps it true" is a real question, not bookkeeping.

The architecture · abstraction

The agent speaks intent, not implementation

What the physicist & agent say

(meta)data_tool“get me this dataset + its cross-section”

transform_tool“turn DAODs into histograms”

batch_tool“run this across the cluster”

inference_tool“fit and get a limit”

What the facility wires underneath (swappable)

rucio-mcp · ami-mcp · Open Data

coffea · uproot · FastFrames · ServiceX · Athena

HTCondor · Dask · Slurm · PanDA · REANA · lxbatch · graphed-org

pyhf · cabinetry · TRExFitter · optimistix · SBI

The agent never names “coffea” or “condor.” It asks for an outcome; the facility decides the backend. That indirection is what makes the same agent portable across facilities.

Second idea. Don't make the agent learn your backends. Give it intent-level tools: a (meta)data_tool to fetch datasets and their metadata (cross-sections), transform_tool, batch_tool, inference_tool. The agent says "get me this dataset and its cross-section" or "histogram these DAODs" — it never says rucio or coffea or condor. Underneath, the facility wires whatever it runs: rucio-mcp/ami-mcp for data; coffea, FastFrames, ServiceX, Athena for transforms; HTCondor, Dask, PanDA, REANA for batch; pyhf or TRExFitter for inference. Different facility, different backends, same high-level tools. This indirection is precisely what lets the same agent move between facilities — the structural basis for the portability claim.

The architecture · portability, concretely

Same engine, different playbook

Only the playbook changes per facility/experiment — the reasoning engine (the “coach”) is untouched. The headline difference is usually data_tool:

playbook/atlas @ UChicago AF

data_tool:   rucio-mcp     # grid
transform:   coffea, FastFrames
batch_tool:  htcondor
inference:   pyhf, TRExFitter

playbook/us-hfcc

data_tool:   eos + https   # browse/dl
transform:   uproot
batch_tool:  local / slurm
inference:   pyhf

The per-tool skills are facility-independent — written once, reused everywhere — so each experiment/facility only writes its thin playbook, not a whole stack. The goal: stop re-developing the same agentic tooling in parallel.

You’ve already met the ATLAS playbook in pieces: the managed CLAUDE.md, the marketplace plugins, the agent’s HTCondor.md.

Concretely, here's a playbook swap. On the left, ATLAS at UChicago: data_tool delegates to rucio-mcp because we're on the grid; transforms via coffea or FastFrames; batch via HTCondor; inference via pyhf or TRExFitter. On the right, a different facility — say a US Higgs-factory effort — whose data_tool today is just "download or browse files off EOS or the web," batch is local or Slurm. Same reasoning engine, same high-level tool names. Only the playbook differs, and the biggest difference is how data_tool is wired. The key reuse point: the skills for each tool are facility-INDEPENDENT — written once, shared — so a new facility only writes a thin playbook, not the whole stack. That's the whole goal: stop every experiment and facility re-developing the same agentic tooling in parallel. And you've already met the ATLAS playbook in pieces — the CLAUDE.md, the marketplace plugins, the agent's memory file. It's just facility context as a versioned artifact.

The portable future · one server, many sites

A real deployment — and a design choice

One hosted rucio-mcp serves several experiments, path-routed — each with its own auth:

rucio-mcp.af.uchicago.edu/site/atlas
rucio-mcp.af.uchicago.edu/site/escape     # also: cms, dune

The choice we made

One MCP, many sites → an analyzer adds rucio-atlas and rucio-escape as separate entries and delegates to each.

The alternative

A separate server per site. Cleaner isolation, but more to host — and FastMCP composition allows only one auth provider per server.

  Tailwind: Anthropic is a DOE Genesis Mission collaborator — MCP servers + skills for science. The AF is a working pilot.

Portability isn't a slide — it's deployed. One hosted rucio-mcp at rucio-mcp.af.uchicago.edu serves several experiments, path-routed: /site/atlas, /site/escape, also cms and dune. Same code, same tools — the playbook differs per site, and each site authenticates separately. But notice this is a DESIGN CHOICE, and it's exactly the granularity question. We chose ONE server serving many sites, so an analyzer configures rucio-atlas and rucio-escape as two entries and delegates to each with its own auth. The alternative is a server per site — cleaner isolation but more to operate, and there's a real constraint: FastMCP's server composition only allows one auth provider per server, which is why something like PandaMCP delegates auth to the PanDA server instead. Hold that for the discussion. And the tailwind: Anthropic joined DOE's Genesis Mission for MCP servers and skills for science — the pattern we built bottom-up, blessed top-down.
