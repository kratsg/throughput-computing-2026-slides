The whole talk in one example

“Watch my HTCondor jobs for me”

A generic LLM

$ watch condor_q

Polls the scheduler every 2 s — for every user who asks. Plausible. Wrong. It hammers the schedd.

An LLM that knows the AF

$ condor_watch_q

The scheduler-friendly tool the facility actually wants you to use — because we told the agent so.

The thesis

The model didn’t get smarter. We gave it facility context. That gap — between a generic assistant and one that knows your storage, scheduler, and data — is the whole game.

This one example is the entire talk. Ask any chatbot to monitor condor jobs and it will confidently tell you to run `watch condor_q`. That's the textbook answer and it's exactly wrong for a shared facility — it pounds the scheduler. The AF agent says `condor_watch_q` instead, because our managed config tells it to. Same model. We just gave it context. Hold onto that: everything else today is variations on "wrap facility knowledge and tools behind the agent."

Grounding — the vocabulary

Four words, one minute

Agent

An LLM in a loop that can call tools, see results, and decide the next step — not just emit text.

MCP server

Model Context Protocol: a standard adapter that exposes a service (Rucio, AMI, your notebook) as tools any agent can call.

Skill

A reusable bundle of instructions + know-how the agent loads on demand (“how we do a pyhf fit here”).

Claude Code

The agent that lives in your terminal — reads files, runs commands, calls MCPs. What we deploy on the login nodes.

Agent

MCP server

Rucio / AMI / Jupyter / …

Fast, because this room is mixed. Agent = LLM that calls tools in a loop. MCP = the USB-C port for tools; write one adapter, every agent can use it. Skill = instructions on demand. Claude Code = the terminal agent we manage on the login nodes. Don't dwell — these are just the nouns for the rest of the talk.

This is the whole thesis

A generic model can’t know that /home is backed up but tiny, that /scratch vanishes, or that XCache exists. We write it down once — the agent inherits it.

  • Storage layout, quotas, backup policy
  • Scheduler etiquette & debugging recipes
  • Data access: XCache, grid proxies, Rucio
  • pixi for new work; how to get help

  Your project CLAUDE.md stacks on top.

This is the slide. Every Claude Code session on the AF silently loads this facility CLAUDE.md alongside the user's own. It's nothing exotic — a storage table, scheduler etiquette, data-access recipes. But it's exactly the stuff a generic model cannot know and will confidently get wrong: that /home is backed up but only 100 GB, that /scratch is node-local and evaporates, that we have XCache. Write the facility down once, every analyzer's agent inherits it. That's the cheapest, highest-leverage thing we did. Everything else is plumbing to deliver context like this.

What you can use today · Jupyter MCP

Drive your AF notebook from anywhere

An MCP server runs inside your JupyterLab pod. Any agent that can reach it — Claude Code on a login node, Claude Desktop, even your phone — can:

  • insert_execute_code_cell — write & run a cell
  • read_cell / read_notebook — see outputs (incl. plots)
  • execute_code — run in the live kernel
  • use_notebook — juggle several notebooks
claude mcp add jupyter --transport http \
  https://jupyterhub.af.uchicago.edu/user/<you>/mcp \
  --header "Authorization: Bearer <token>"

Token from /hub/token (scope access:servers); lifetime tied to your server session.

What you can use today · agents on your behalf

An agent watches the cluster for you

A scheduled agent scans HTCondor daily, posts a cluster report — and when your jobs are stuck, drafts you a specific fix:

“56 jobs held: out of memory — used 32,694 MB against a 32,768 MB limit. Bump request_memory to ~36,000. Tip: write large outputs to /data, not /home.”

  Human-in-the-loop: emails are drafted, approved in Slack, then sent — never autonomously.

It only gives good advice because it carries AF knowledge (HTCondor.md):

Hold reasonThe fix it knows
OOM (cgroup)Raise memory request
errno 122 disk quotaFree space → rm → resubmit
errno 13 on c111/c113Sysadmin issue — don’t email user
wall-time > 4h (short)Split into shorter chunks

  Same lesson as the facility CLAUDE.md: encoded knowledge → useful agent.

Deeper on the ops machinery (Shannon/OpenClaw, multi-agent fleet) → next talk, “Towards Agentic Ops.”
Some agents work even when you're not there. A scheduled agent scans HTCondor every morning, posts a cluster report, and — the user-facing part — when YOUR jobs are held, it drafts you an email with the specific fix. Real example: 56 OOM jobs, it read the actual numbers, 32,694 against a 32,768 limit, and said bump to 36,000 and write to /data. Crucial: it does NOT send autonomously. Drafts go to Slack, a human approves, then it sends. We even hit a race condition where two agents both grabbed approvals — so we stood email-sending down until fixed. I'd rather be honest about that than pretend it's flawless. Every win today was the middle term, not the model: the managed CLAUDE.md, the marketplace plugins, the MCPs over Rucio/AMI, the condor bot’s HTCondor.md. Swap in next year’s smarter model — the context is still what makes it ours.

If you take one slide home, this one. The frontier model is a commodity — everyone has the same Claude or GPT. What you can't buy off the shelf is the context: how YOUR facility is laid out, which tools matter, how ATLAS does things. Look back at section one — every single win was the middle term. The CLAUDE.md, the marketplace, the MCPs, the agent's memory file. Not the model. Corollary: the model will keep getting better on its own. Our job is the context. That's also why this is a facility conversation, not just a vendor one.

The architecture · abstraction

The agent speaks intent, not implementation

What the physicist & agent say

transform_tool“turn these DAODs into histograms”

batch_tool“run this across the cluster”

inference_tool“fit and get a limit”

What the facility wires underneath (swappable)

coffea · uproot · FastFrames · Athena EventLoop

HTCondor · Dask · Slurm · PanDA

pyhf · cabinetry · TRExFitter

The agent never names “coffea” or “condor.” It asks for an outcome; the facility decides the backend. That indirection is what makes the same agent portable across facilities.

Second idea. Don't make the agent learn your backends. Give it intent-level tools: transform_tool, batch_tool, inference_tool. The agent says "histogram these DAODs" or "fit this" — it never says coffea or condor. Underneath, the facility wires whatever it runs: coffea or FastFrames or Athena; condor or Dask or PanDA; pyhf or TRExFitter. Different facility, different backends, same high-level tools. This indirection is precisely what lets the same agent move between facilities. It's the structural basis for the portability claim I'm about to make.

The architecture · separation of concerns

Codec vs codebook

  Codec — the framework

Model-agnostic, experiment-agnostic. Orchestration, tool routing, execution, guardrails.

sharedwritten once

  Codebook — the context

Everything facility- and experiment-specific:

  • tools.yaml — which MCPs & backends
  • CLAUDE.md / system prompt — the AF
  • knowledge corpus & few-shot examples

swappable

codebook/atlas-uchicago

codebook/eic

Swap the codebook, keep the codec — port the whole assistant to another experiment or facility.

You’ve already seen the codebook: the managed CLAUDE.md, the marketplace plugins, the agent’s HTCondor.md. In practice, the codebook is just the facility context, made into an artifact.

Third idea, and it names what we've been circling. Split the system in two. The codec is the framework — orchestration, routing, guardrails — model- and experiment-agnostic, written once. The codebook is all the context: which tools, the system prompt, the knowledge, the examples. Portability becomes concrete: swap codebook/atlas-uchicago for codebook/eic and the same codec runs a different experiment. And here's the punchline tying the whole talk together — you've already met the codebook. The managed CLAUDE.md, the marketplace plugins, the agent's memory file. The codebook is just facility context turned into a versioned artifact.

The portable future · proof

Same tool, different facility — today

rucio-mcp already serves two entirely different Rucio infrastructures — the codebook is one flag:

ATLAS

pixi exec rucio-mcp serve \
  --site atlas --read-only

ESCAPE

pixi exec rucio-mcp serve \
  --site escape --read-only

Identical agent, identical tools. Swap the context, not the code — exactly the codec/codebook split, proven on real infrastructure.

  Tailwind: Anthropic is a DOE Genesis Mission collaborator — MCP servers + skills for science. The AF is a working pilot of exactly that pattern.

Is portability real or a slide? Here's the proof. The same rucio-mcp already serves two completely different Rucio instances — ATLAS and ESCAPE — and the difference is one flag, --site. Same agent, same tools, swap the context. That IS the codec/codebook split, running on real infrastructure today, not a promise. And there's a tailwind: Anthropic joined DOE's Genesis Mission committing to MCP servers and skills for science. The pattern we built bottom-up is the same one being blessed top-down — the AF is a working pilot.

Let’s discuss — the open questions

What should we decide together?

  Granularity & identity

One MCP per service, or a single mcp.af.uchicago.edu gateway delegating identity? Who owns the token sprawl?

  Who owns the codebook?

Each facility writes its own context — but who maintains it, reviews it, keeps it true?

  What may an agent do?

Read freely — sure. Write, submit, delete, email? Where’s the human-in-the-loop line?

  Who pays for inference?

At HL-LHC scale: hosted frontier models vs self-hosted open weights on facility GPUs?

  One marketplace for all?

A shared USATLAS marketplace + codebook across every AF — who curates the commons?

My bet: the model is solved for us. Context, identity, and trust are the facility’s homework.

This is what I actually want from the next 10 minutes. Five questions, none of which a vendor answers for us. Granularity and identity: do we run an MCP per service, or a single gateway that delegates identity — and how do we stop users drowning in tokens? Who maintains each facility's codebook so it stays true? What is an agent allowed to DO without a human — reads are easy, writes and emails are not? Who pays for inference at HL-LHC scale, hosted vs self-hosted on our own GPUs? And should there be ONE USATLAS marketplace and codebook across all AFs, and who curates that commons? My bet, to provoke you: the model is the solved part. Context, identity, and trust are our homework. Let's argue about it.
